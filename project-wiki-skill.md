# Vibe Coding 项目的 Wiki 技能

> 人类 + AI Agent 协作开发时，如何组织项目 wiki 让 Agent 输出质量最高。
> 适用于任何使用 Claude Code（或类似 AI 编程助手）的产品项目。

---

## 背景

Vibe Coding 的核心特征：**你不逐行写代码，你描述意图，Agent 写代码。**

这改变了"文档"的角色。传统开发中文档是写给人看的参考资料，经常滞后于代码。Vibe Coding 中文档是写给 Agent 看的输入物料——它直接决定 Agent 的输出质量。

一个没有 wiki 的 Vibe Coding 项目，每次对话都在从零开始对齐上下文。你会发现自己反复解释同样的事情："我们用的是 Express 不是 Fastify""这个项目的设计语言是极简风""某代理和标准 API 有兼容性问题"。

---

## 文档体系：全局 + 阶段模型

Vibe Coding 项目的文档自然分为两个维度：**内容层次**（Why / What / How）和**粒度层次**（全局 / 阶段）。

### 内容层次

| 层次 | 回答的问题 | 读者 |
|------|-----------|------|
| **Why** — 产品与决策 | 做什么？不做什么？为什么这样选？ | 你自己 + Agent |
| **What** — 功能规划 | 做了哪些功能？状态如何？设计取舍？ | Agent（开始新功能前读） |
| **How** — 代码参考 | 架构长什么样？接口是什么？数据流？ | Agent（修改具体文件时查） |

### 粒度层次：全局 vs 阶段

| | 全局文件（稳定，新阶段才改） | 阶段文件（增量更新，功能实现时改） |
|---|---|---|
| **Why** | `product-proposal.md` | — |
| **What** | `project-roadmap.md` — 功能索引鸟瞰 | `stage-X.md` — 设计决策 |
| **How** | `system-architecture.md` — 架构 + 类型 + API 路由表 | `stage-X.md` — 功能特有的 API、数据模型、受影响文件 |

**关键洞察：stage 文件同时包含 What 和 How。** 一个功能的设计决策（What）和它的 API 契约、数据模型（How）天然绑定在一起。把它们拆到不同文件会导致 Agent 要同时打开两个文件才能理解一个功能。stage 文件是功能的完整上下文，全局文件是跨功能的导航索引。

**分工原则：**
- 全局 How（`system-architecture.md`）记录**稳定的、跨功能的**信息——目录结构、数据流、设计系统、核心类型定义、完整 API 路由表
- Stage 文件记录**功能特有的** How 细节——这个功能新增了什么 API？数据模型长什么样？影响哪些文件？
- 新功能交付后，新增的 API 路由追加到 `system-architecture.md` 的路由表（一行即可），详细的请求/响应格式留在 stage 文件中

## 哪些文件值得维护

### 第一梯队：必须有，直接影响 Agent 输出质量

#### 1. `product-proposal.md` — 产品提案

**写给谁看：** Agent + 未来的你

**核心内容：**
- 产品定位（一句话说清楚做什么）
- 不做什么（这条比"做什么"更重要——防止 Agent 越界发挥）
- 目标用户和核心场景
- 功能矩阵（功能 × 解决的问题 × 阶段）
- 路线图叙事（各阶段一句话目标 + 成功标志）

**为什么必须有：** Agent 没有产品直觉。你不告诉它边界在哪，它就会在你让它加个简单功能时顺手加上一堆你不需要的东西。产品提案是 Agent 的行为边界。

**维护频率：** 每个大阶段开始前检查一次，通常改动很小。

#### 2. `project-roadmap.md` — 功能索引

**写给谁看：** Agent + 你自己

**核心内容：**
- 阶段总览表（阶段 × 主题 × 状态 × 链接到 stage 文件）
- 全量功能索引（所有功能按阶段分组，每行标注状态）
- 关键里程碑

**为什么必须有：** Agent 需要一个"鸟瞰图"来理解项目全貌。product-proposal 的路线图是叙事性的（讲为什么），roadmap 是索引性的（查什么功能在哪、状态如何）。两者互补不重叠。

**维护频率：** 新功能定义时追加行，阶段完成时批量更新状态。改动频率高于 product-proposal，低于 stage 文件——这正是它独立存在的原因。

#### 3. `system-architecture.md` — 系统架构

**写给谁看：** Agent

**核心内容：**
- 技术栈概览、架构图
- 目录结构
- 数据流（请求链路、推送机制、定时任务等）
- 设计系统（颜色、动画、状态映射——如果有前端）
- 核心类型定义（跨功能的 domain types）
- 完整 API 路由表（方法、路径、响应类型）
- 文件约定、环境变量、构建命令

**为什么必须有：** Agent 动手前先读这个。它回答"项目长什么样"，帮 Agent 建立全局心智模型。没有它，Agent 要读完所有源文件才能定位该改哪里。

**控制长度：** 建议 300-500 行。超出就说明混入了功能特有的细节，应该拆到 stage 文件。

**维护频率：** 架构变更时更新（新增模块、数据流改变、新 API 路由）。Bug fix 不用更新。

---

### 第二梯队：按需编写，提升特定阶段效率

#### 4. `stage-X.md` — 阶段文件（What + How 合体）

**写给谁看：** Agent

**核心内容：**
- 功能汇总表（功能 × 状态）
- 每个功能的：用户场景、设计决策与取舍
- 每个功能的：API 契约、数据模型、受影响文件（How 细节）
- 实现摘要（已完成时）
- 遗留项 / Backlog

**什么时候写：** 功能复杂度超过"一句话能说清"的时候。

**反模式警告：** 不要把 roadmap（功能状态表）和 design-spec（设计规格）拆成两个文件维护。实践中必然出现：
- 两个文件记录同一个功能的状态，经常不一致
- Agent 要同时打开两个文件才能理解一个功能的全貌
- 功能交付后，design-spec 中的验收清单变成死代码

统一为 `stage-X.md`：一个文件里既有状态表格，也有设计决策和 API 细节。**一个功能的所有信息在一个地方。**

**建议每份 stage 文件控制在 150-300 行。** Agent 真正需要的是数据模型、API 契约和文件影响范围，UI 交互细节它自己能处理得很好。

#### 5. 外部系统参考

**写给谁看：** Agent

**什么时候写：** 项目依赖外部系统的私有数据格式时（第三方 API 的 undocumented 行为、私有文件格式等）。不记下来的话，每次新对话都得重新分析一遍。

**维护频率：** 外部系统更新时才需要改。

#### 6. `known-pitfalls.md` — 踩坑记录

**写给谁看：** Agent + 未来的你

**核心内容：**
- 按类别组织（API 集成、文件系统、并发、前端等）
- 每条记录三要素：现象、原因、解决方案

**为什么需要：** Agent 不会自动记住上一次对话中踩过的坑。pitfalls 是"动手前先扫一遍"的预防性文档，和 stage 文件的"设计决策"互补——stage 记录"选了什么"，pitfalls 记录"为什么不能选另一个"。

**维护频率：** 踩一次记一次。不用等阶段结束。

#### 7. `CHANGELOG.md` — 变更日志

**写给谁看：** 你自己

**为什么需要：** Vibe Coding 的 git log 通常很粗，因为你不会花时间写精细的 commit message。changelog 是你补上这个信息差的地方——记录"什么时候做了什么，为什么这样做"。

**维护频率：** 每个阶段交付后补一笔。不用每次 commit 都写。

#### 8. `development-guide.md` — 开发指南

**写给谁看：** 未来的你 / 协作者

**什么时候写：** 项目有非显而易见的配置要求时（代理地址、特殊环境变量、构建顺序依赖）。如果 `npm install && npm run dev` 就能跑起来，可以不写。

---

### 不需要维护的

| 文档类型 | 为什么不需要 |
|---------|-------------|
| API 文档（Swagger/OpenAPI） | system-architecture.md 里的路由表 + stage 文件的详细契约够用了 |
| 组件文档（Storybook） | Vibe Coding 项目的 UI 迭代太快，独立组件文档瞬间过时 |
| 会议纪要 | 你的"会议"就是对话历史，Agent 的 MEMORY.md 比纪要更有效 |
| 贡献指南（CONTRIBUTING.md） | 除非你真的有其他人类贡献者 |
| 架构决策记录（ADR） | 小项目用 stage-X.md 里的"设计决策"章节就够了 |

## 推荐 Wiki 结构

```
wiki/
├── product-proposal.md       # Why — 产品提案：愿景、边界、场景
├── project-roadmap.md        # What（全局）— 跨阶段功能索引、状态追踪
├── stage-a.md ~ stage-N.md   # What+How（阶段）— 设计决策 + API + 数据模型 + 实现摘要
├── system-architecture.md    # How（全局）— 架构 + 类型 + API 路由表 + 设计系统
├── known-pitfalls.md         # 踩坑记录：现象 → 原因 → 解决方案
├── CHANGELOG.md              # 变更日志 + 关键决策
└── development-guide.md      # 环境搭建、构建部署、常见问题（可选）
```

按需追加：外部系统参考（`external-*.md`）、其他项目特有的参考文档。

全局文件提供跨功能的导航索引（稳定），阶段文件提供功能的完整上下文——What 和 How 合体（增量更新）。

## 实操建议

### 什么时候写文档

| 时机 | 写什么 |
|------|--------|
| 项目启动 | `product-proposal.md`（含路线图叙事） |
| 第一个阶段交付后 | `system-architecture.md`（此时架构已稳定） |
| 开始新阶段前 | `project-roadmap.md` 追加功能行 + `stage-X.md`（如果功能复杂） |
| 每个阶段交付后 | 更新 `stage-X.md` 状态 + `project-roadmap.md` 状态 + `CHANGELOG.md` |
| 踩到技术坑 | 记到 `known-pitfalls.md`（现象 → 原因 → 解决方案） |
| 新增 API 路由后 | `system-architecture.md` 路由表追加一行，详细契约在 stage 文件 |

### 写文档的原则

1. **写给 Agent 看的文档，用结构化格式。** 表格 > 段落，代码块 > 文字描述，类型定义 > 自然语言解释。Agent 解析表格和代码块的准确率远高于读散文。

2. **写给自己看的文档，记决策不记细节。** "为什么选了 X 而不是 Y"比"X 的安装命令是什么"有价值得多。细节可以搜，决策上下文会丢。

3. **不要重复。** 一个信息只在一个地方维护。两个地方记录同一个信息——必然不一致。

4. **Stage 文件是功能的唯一权威来源。** What（设计决策）和 How（API、数据模型）放在一起。全局文件只做索引和导航，不重复 stage 的细节。

5. **允许过时，但标记过时。** 文档 100% 同步是幻想。接受某些细节会落后于代码，但在文件头部用版本号和日期标记，让读者（包括 Agent）知道这份文档的新鲜度。

6. **宁可不写，不要写了不维护。** 一份过时的 design-spec 比没有 design-spec 更危险——Agent 会照着错误的规格写代码，你 debug 半天才发现是文档的问题。

7. **文件数量不是问题，信息重叠才是。** 十几个文件只要每个 100-500 行、职责单一，比 3 个各 2000 行、彼此重叠的文件好维护得多。

### 文档维护的投入比例

整个项目生命周期中，文档维护时间应占 **10-15%**。低于 10% 说明你在每次对话中花太多时间重复对齐上下文；高于 15% 说明你在过度文档化，Vibe Coding 的速度优势被消耗了。

## 真正发挥作用的模式

1. **system-architecture 避免"Agent 改错文件"。** 没有它的时候，Agent 经常在入口文件里堆代码；有了它之后，Agent 知道该改哪个模块。

2. **product-proposal 的"不做什么"拦住 Agent 的过度发挥。** 明确说了边界，Agent 就不会在简单功能上顺手加一堆你不要的东西。

3. **外部系统参考省大量逆向工程时间。** 私有格式没有官方文档，写一次后面所有阶段都在引用。

4. **Stage 文件中 What + How 合体是最高效的组织方式。** Agent 开发一个功能只需读一个文件，设计决策、API 契约、数据模型、受影响文件全在一处。

5. **project-roadmap 的鸟瞰索引让 Agent 快速定位。** "这个功能在哪个 stage？状态如何？" 一张表回答，不用翻所有 stage 文件。

6. **合并 roadmap + design-spec → stage-X.md 消除信息不一致。** 两个文件记录同一个功能的状态，一个说"已完成"另一个还写着验收清单——这种矛盾让 Agent 和人类都困惑。

7. **known-pitfalls 阻止 Agent 重复踩坑。** Agent 不会记住上次对话的失败尝试，pitfalls 文件让它动手前就知道哪些路走不通。

# evo-agent-team · 产品方案

> **版本：** v2.0 · 2026-03-06
> **状态：** Stage A 已交付，Stage B 规划中

---

## 一、产品愿景

### 我们要做什么

一个 AI 编程团队的监控与管理平台，让开发者能够 **看见、理解、纠正、积累** 团队的工作过程和成果。

### 核心信念

1. **AI 团队应该像真人团队一样可管理。** 你能看到谁在做什么、进展如何、卡在哪里。
2. **每一次纠正都是一笔投资。** 你给出的反馈不会消失，而是沉淀为团队记忆，同样的错误不犯第二次。
3. **人类保留最终决策权。** 系统在合适的时机主动请示，而不是让你不停地盯着屏幕。

### 一句话定位

**让你的 AI 编程团队持续进化，更加专业。**

### 产品不做什么

- 不做 AI 代码编辑器（不替代 Cursor、Copilot）
- 不做项目管理工具（不替代 Linear、Jira）
- 不做模型训练平台（不涉及 RLHF、DPO、微调）
- 不做通用 AI Agent 框架（只服务 Claude Code 团队模式）

## 二、目标用户

### 用户画像

| 维度 | 描述 |
|------|------|
| **角色** | 全栈开发者 / 技术负责人 / 独立开发者 |
| **技术水平** | 熟悉命令行，已在使用 Claude Code |
| **团队规模** | 1 人使用 Claude Code 的多 Agent 团队模式（3-8 个 Agent 并行） |
| **工作模式** | 启动 Agent 团队后，希望在团队运行期间和运行结束后进行监控和评审 |

### 用户当前的替代方案

| 替代方案 | 体验 |
|---------|------|
| Claude Code CLI 原生输出 | 只看得到 Lead 的日志流，Teammate 行为完全不可见 |
| 手动翻看 `~/.claude/` 文件 | 需要理解 JSONL 格式，极其低效 |
| tmux 多窗口 | 信息分散，无法汇聚全局视图 |

### 用户核心诉求（按优先级）

1. **看清全局** — 团队在做什么？进展到哪了？
2. **及时干预** — 谁卡住了？需要我做什么？
3. **积累经验** — 我的纠正能不能被记住？下次别再犯。
4. **降低成本** — Token 花了多少？哪里浪费了？

## 三、现有方案的局限性

> 以下问题基于 Claude Code 多 Agent 团队模式的实际使用观察。这些局限性并非 Claude Code 的设计缺陷，而是 CLI 工具的天然边界 — 它专注于推理和执行，不负责团队的长期管理和人类协作。

### 3.1 黑盒运行 — "不知道它们在干什么"

| 现象 | 影响 |
|------|------|
| 只能看到 Lead Agent 的日志流 | Teammate 在后台做了什么完全不可见 |
| 多 Agent 日志交织 | 分不清哪条日志属于哪个 Agent |
| 没有全局视图 | 无法一眼看到"5 个 Agent 中有 2 个卡住了" |

**用户真实反应：** "我启动了一个 5 人团队，跑了 20 分钟，回来发现 3 个 Agent 一直在等我确认，但我完全不知道。"

### 3.2 干预真空 — "知道出了问题但做不了什么"

| 现象 | 影响 |
|------|------|
| Agent 请求确认只能在 CLI 中响应 | 必须找到正确的终端窗口，跳回 CLI 操作 |
| 没有告警机制 | Agent 卡在循环中 10 分钟，用户无感知 |
| 没有关键路径分析 | 不知道哪个任务阻塞了所有下游 |

**用户真实反应：** "Agent 陷入了死循环，Token 一直在烧，我过了半小时才注意到。"

### 3.3 经验蒸发 — "每次都从零开始"

| 现象 | 影响 |
|------|------|
| 团队即用即毁 | 上次团队的经验无法带到下次 |
| 反馈没有沉淀通道 | 用户纠正了一个错误，下次同样的错误还会犯 |
| MEMORY.md 靠手动维护 | 用户需要自己总结和编辑记忆文件 |

**用户真实反应：** "我告诉它'不要用 try-catch，用 Result 类型'，下一个会话它又用回去了。"

### 3.4 监督疲劳 — "要么不管，要么累死"

| 现象 | 影响 |
|------|------|
| 只有"全自动"和"逐步确认"两种模式 | 缺少中间地带 — "重要的事问我，小事自己决定" |
| Agent 频繁请求低价值确认 | 用户被打断太多次后关闭自主权限 |
| 无法针对特定操作设置审批 | 要么全放开，要么全拦截 |

**用户真实反应：** "它每次读文件都问我一遍 yes/no，问了 30 次之后我直接把权限全开了。然后它删了一个不该删的文件。"

### 3.5 成本黑洞 — "不知道钱花在哪了"

| 现象 | 影响 |
|------|------|
| Token 消耗数据散落在 JSONL 文件中 | 无法汇总看到团队总消耗 |
| 无法按 Agent 或工具归因 | 不知道哪个 Agent 最"贵" |
| 无法检测异常消耗 | Agent 循环调用工具时 Token 静默流失 |

**用户真实反应：** "跑了一个小时的任务，结束后发现花了 50 万 Token，但不知道是哪个环节的问题。"

### 局限性小结

| 局限性 | 根因 | 我们的解法 |
|--------|------|-----------|
| 黑盒运行 | CLI 只展示 Lead 日志流 | Dashboard + 拓扑图 + 通信日志 (Stage A) |
| 干预真空 | 没有 UI 交互层 | 智能告警 (Stage B) + UI 内响应 (Stage C) |
| 经验蒸发 | 没有反馈沉淀机制 | 反馈标注 + 记忆管理 + 指南演化 (Stage D) |
| 监督疲劳 | 缺少精细化权限控制 | 规则化监督 + 偏好自动沉淀 (Stage E) |
| 成本黑洞 | Token 数据无可视化 | 成本分析面板 (Stage B) |

## 四、用户场景

### 场景 1：晨会检查 — "昨晚跑的怎么样？"

> **触发时机：** 用户启动 Agent 团队后离开，隔几小时回来
> **当前体验：** 需要翻看 CLI 输出、手动检查任务文件，逐个确认进度
> **期望体验：** 打开 Dashboard，2 分钟内了解全局

**用户需要知道的信息：**
- 完成了哪些任务？
- 当前卡在哪里？卡了多久？
- 有没有需要我做决定的事？
- 花了多少 Token？

### 场景 2：实时监控 — "它们现在在干嘛？"

> **触发时机：** Agent 团队正在运行中
> **当前体验：** Lead 日志不断滚动，无法区分不同 Agent 的行为
> **期望体验：** 可视化地看到每个 Agent 的状态和任务进度

**用户需要看到的信息：**
- 每个 Agent 当前在做什么任务
- Agent 之间的消息传递（谁在和谁沟通）
- 任务依赖关系图（哪个任务阻塞了下游）
- 哪个 Agent 在等我确认

### 场景 3：紧急干预 — "这个 Agent 好像卡住了"

> **触发时机：** 某个 Agent 长时间无进展，或 Agent 请求人工确认
> **当前体验：** 需要切回 CLI 终端操作
> **期望体验：** 直接在 Dashboard 中响应 Agent 的问题

**用户需要做的操作：**
- 看到告警（谁卡住了、卡了多久）
- 查看 Agent 的具体提问
- 在 UI 中输入回复
- 确认 Agent 恢复工作

### 场景 4：复盘评审 — "这段代码写得不对"

> **触发时机：** 团队运行结束后，用户检查产出质量
> **当前体验：** 在编辑器中逐文件 review，发现问题只能心里记着
> **期望体验：** 在会话历史中标注反馈，系统自动记住

**用户需要做的操作：**
- 浏览某个 Agent 的完整会话历史（不只是 Lead 的）
- 对某条输出标注 👍（好的做法）或 👎（需要纠正，附理据）
- 反馈自动沉淀为团队知识，下次生效

### 场景 5：新项目启动 — "用上次的经验"

> **触发时机：** 启动新项目，想复用老团队积累的经验
> **当前体验：** 手动复制 TEAM_GUIDE.md，逐条回忆哪些规则适用
> **期望体验：** 系统推荐可迁移的知识，一键导入

**用户需要做的操作：**
- 选择"从哪个老团队导入经验"
- 审核推荐的知识条目（区分通用 vs 项目专属 vs 临时修复）
- 确认导入

### 场景 6：成本管控 — "花了这么多 Token？"

> **触发时机：** 团队运行结束后查看消耗
> **当前体验：** Token 数据散落在 JSONL 文件的 `usage` 字段中，无法汇总
> **期望体验：** 一个成本面板，按 Agent、按工具、按时间展示消耗

**用户需要看到的信息：**
- 团队总 Token 消耗
- 哪个 Agent 消耗最多
- 哪些工具调用最"贵"
- 有没有异常消耗（如 Agent 在循环）

## 五、产品方案

### 产品架构

```
┌──────────────────────────────────────────────────────┐
│                    用户界面层                          │
│  Dashboard · 拓扑图 · 通信日志 · 时间线 · 会话历史     │
│  成本面板 · 反馈标注 · 记忆管理 · 监督规则配置         │
├──────────────────────────────────────────────────────┤
│                    智能服务层                          │
│  执行摘要生成 · 告警引擎 · 偏好提取 · 语境摘要更新      │
├──────────────────────────────────────────────────────┤
│                    数据接入层                          │
│  读取 ~/.claude/ 目录 · 解析 JSONL · 文件监听          │
│  有限写入：反馈日志 · 记忆文件 · 监督规则               │
├──────────────────────────────────────────────────────┤
│                Claude Code CLI（外部系统）              │
│  Agent 团队运行时 · 文件系统存储 · 消息传递              │
└──────────────────────────────────────────────────────┘
```

**关键架构原则：**
- **读多写少：** 主要读取 Claude Code 产生的文件，仅对反馈、记忆和监督规则做有限写入
- **降级可用：** AI 功能（摘要、偏好提取）不可用时，自动降级为纯数据展示
- **本地运行：** 不依赖外部服务，数据不离开本机

### 功能矩阵

以下按"解决的用户场景"组织功能，而非按技术模块：

#### 看清全局 — 让团队透明

| 功能 | 解决什么问题 | 阶段 |
|------|-------------|------|
| 团队仪表板 | 一屏看到团队概览、成员状态、任务进度 | A ✅ |
| 任务拓扑图 | 看到任务之间的依赖关系，识别关键路径 | A ✅ |
| 通信日志 | 看到 Agent 之间在说什么，按类型和发送者过滤 | A ✅ |
| 时间线 | 看到任务状态变更的完整历史 | A ✅ |
| 执行摘要 | 2 分钟了解"团队做了什么、卡在哪、关键决策" | B |
| 跨 Agent 会话浏览 | 不只看 Lead，还能看每个 Teammate 的详细行为 | B |

#### 及时干预 — 让问题不被忽略

| 功能 | 解决什么问题 | 阶段 |
|------|-------------|------|
| 人工输入提醒 | 检测到 Agent 在等确认时，顶部显示告警 | A ✅ |
| 智能告警 | Agent 卡住、Token 异常、关键路径阻塞时主动提醒 | B |
| UI 内响应 | 直接在 Dashboard 中回复 Agent 的问题，不用切终端 | C |
| 任务状态编辑 | 在 UI 中直接修正任务状态 | C |
| 监督规则配置 | 定义哪些操作必须经过人工审批 | E |

#### 积累经验 — 让纠正有价值

| 功能 | 解决什么问题 | 阶段 |
|------|-------------|------|
| 反馈标注 | 在会话历史中对 Agent 输出进行 👍/👎/📌 评价 | D |
| Agent 成长档案 | 查看每个 Agent 的任务历史、成功率、收到的评价 | D |
| 团队指南演化 | 反馈自动归纳为团队规则，用户审批后生效 | D |
| 偏好沉淀 | 高频反馈自动注入 Agent 行为，减少重复纠正 | E |
| 跨团队知识迁移 | 把老团队的经验带到新项目 | E |

#### 降低成本 — 让消耗可控

| 功能 | 解决什么问题 | 阶段 |
|------|-------------|------|
| Agent Token 统计 | 在成员卡片上展示每个 Agent 的 Token 消耗 | A (backlog) |
| 成本分析面板 | 按 Agent、按工具、按时间展示 Token 消耗分布 | B |

#### 保持一致 — 让协作不出岔子

| 功能 | 解决什么问题 | 阶段 |
|------|-------------|------|
| 冲突检测 | 多 Agent 改同一文件时，检测并展示差异 | D |
| 会话记忆管理 | 管理项目记忆文件，确保新 Agent 继承关键上下文 | D |
| 语境摘要注入 | 新 Teammate 自动获得 Lead 的关键决策摘要 | E |

## 六、路线图

> 各阶段详细功能定义见 `wikis/project-roadmap.md`。

### Stage A：可视化（已完成）

**解决的核心问题：** Agent 团队是黑盒，用户看不到发生了什么。

**交付成果：** 五大视图（仪表板、拓扑图、通信日志、时间线、会话历史）+ 实时推送 + 导出功能 + Demo 模式。

**结论：** 用户能看到团队在做什么了。但"看到"不等于"看懂"— 原始数据量太大，用户仍然需要花很多时间消化信息。

### Stage B：智能观测

**解决的核心问题：** 信息过载 — 用户看到了数据但抓不住重点。

**关键交付物：**
- 执行摘要 — 一段话总结团队进展（对应场景 1：晨会检查）
- 智能告警 — 系统主动告诉你哪里出了问题（对应场景 3：紧急干预）
- 跨 Agent 会话浏览 — 能看每个 Teammate 的详细行为（对应场景 2：实时监控）
- 成本分析 — Token 消耗一目了然（对应场景 6：成本管控）

**成功标志：** 用户从"打开 Dashboard 需要 10 分钟才能搞清状况"变为"2 分钟看完摘要就够了"。

### Stage C：轻量干预

**解决的核心问题：** 知道出了问题但做不了什么 — 用户必须切回终端才能操作。

**关键交付物：**
- UI 内响应 — 不用切终端就能回复 Agent（对应场景 3：紧急干预）
- 任务状态编辑 — 在 UI 中直接修正任务数据（对应场景 2：实时监控）

**成功标志：** 用户在 UI 中响应 Agent 请求的比例超过在终端中响应的比例。

### Stage D：持久化与反馈闭环

**解决的核心问题：** 每次都是从零开始 — 用户的纠正没有被记住。

**关键交付物：**
- 反馈标注系统 — 在会话中给 Agent 打分和写评价（对应场景 4：复盘评审）
- Agent 成长档案 — 记录每个 Agent 的经验和评价（对应场景 4：复盘评审）
- 团队指南自动演化 — 反馈归纳为规则，审批后生效（对应场景 4：复盘评审）
- 会话记忆管理 — 确保新 Agent 知道项目背景（对应场景 5：新项目启动）

**成功标志：** 用户在新会话中的纠偏频率开始下降 — 团队真的在"记住"。

### Stage E：记忆驱动的演化

**解决的核心问题：** 规则靠人维护太累 — 系统应该主动从反馈中学习。

**关键交付物：**
- 偏好自动沉淀 — 高频反馈自动提升为行为规则（对应场景 4 的延伸）
- 规则化按需监督 — 系统只在"真正需要人"时才打断（对应场景 3 的延伸）
- 跨团队知识迁移 — 老团队经验一键带到新项目（对应场景 5：新项目启动）
- 语境摘要注入 — 新 Teammate 自动获得背景信息（对应场景 2 的延伸）

**成功标志：** 用户感受到"这个团队越来越懂我了"— 纠偏越来越少，主动请示越来越精准。

## 七、成功指标

### 北极星指标

**用户在新会话中的纠偏频率**（纠偏次数 / Agent 输出次数）

这个指标直接衡量了产品的核心价值 — "团队是否在进化"。如果纠偏频率持续下降，说明反馈沉淀机制在起作用。

### 各阶段关键指标

| 阶段 | 指标 | 衡量方式 | 目标 |
|------|------|---------|------|
| **A** | 视图覆盖率 | 所有视图能正常展示真实团队数据 | 5/5 视图可用 |
| **A** | 数据刷新延迟 | 从文件变更到 UI 更新 | < 4 秒 |
| **B** | 摘要有效性 | 用户是否阅读摘要后不再翻原始日志 | > 70% 会话仅看摘要 |
| **B** | 告警精准率 | 告警中被用户响应（而非关闭）的比例 | > 60% |
| **C** | 干预响应位置 | 用户在 UI 中响应（vs 切回终端）的比例 | > 50% |
| **D** | 反馈标注率 | 用户对 Agent 输出进行评价的比例 | > 10% 的 assistant 消息 |
| **D** | 指南采纳率 | 系统建议的 TEAM_GUIDE 更新被用户采纳的比例 | > 60% |
| **E** | 纠偏频率下降 | 相比 Stage D 初期，用户纠偏频率的变化 | 下降 > 30% |
| **E** | 监督精准度 | 系统请求人工审批时，用户确实做出修改的比例 | > 50% |

### 反向指标（护栏）

| 指标 | 含义 | 红线 |
|------|------|------|
| Token 开销增幅 | 平台自身的 AI 功能（摘要、偏好提取等）消耗的额外 Token | 不超过团队本身消耗的 15% |
| 系统延迟 | 平台对 Agent 团队运行速度的影响 | 增加延迟 < 200ms |
| 记忆文件大小 | TEAM_GUIDE.md + MEMORY.md 的总长度 | 不超过 8K Token |
| 监督请求频率 | 系统主动打断用户的次数 | 每小时不超过 3 次 |

## 八、竞争格局与差异化

### 竞品对比

| 维度 | Claude Code 原生 | Cursor / Copilot | Devin | 本产品 |
|------|-----------------|-----------------|-------|--------|
| 多 Agent 协作 | ✅ 支持 | ❌ 单 Agent | ✅ 支持 | ✅ 基于 Claude Code |
| 运行可视化 | ❌ CLI 日志流 | N/A | ✅ 有 Web UI | ✅ 五大视图 + 拓扑图 |
| 人工干预 | ⚠️ 仅 CLI | ⚠️ 内联 | ⚠️ 有限 | ✅ UI 内响应 + 规则化监督 |
| 反馈沉淀 | ❌ 无 | ❌ 无 | ❌ 无 | ✅ 反馈 → 记忆 → 行为变化 |
| 跨会话记忆 | ⚠️ MEMORY.md（手动） | ❌ 无 | ❌ 无 | ✅ 记忆管理 + 自动演化 |
| 成本可视化 | ❌ 无 | ❌ 无 | ⚠️ 有限 | ✅ 按 Agent/工具/时间 |

### 核心差异化

**我们不做另一个 AI 编码工具，我们做 AI 编程团队的"管理平台"。**

差异化壁垒在于**反馈飞轮**：用户纠正 → 沉淀为记忆 → Agent 行为改善 → 纠正减少 → 用户更愿意使用 → 更多反馈。用得越久，团队越专业，迁移成本越高。
